# 003：区分敏捷与瀑布模型 🏗️➡️🔄

![](img/9c4b787b2d995fdd5f625522ce6b2907_0.png)

在本节课中，我们将深入探讨敏捷项目管理与瀑布模型之间的核心区别。理解这些差异是掌握敏捷思维的基础。

在上一节视频中，我们简要回顾了敏捷的历史并介绍了《敏捷宣言》。在详细解读宣言的价值观和原则之前，让我们花更多时间比较敏捷与瀑布模型。本节将重点阐述区分敏捷与瀑布的关键要素，这将为我们后续深入学习《敏捷宣言》打下坚实基础。

正如历史课中提到的，敏捷是为了应对瀑布模型严格线性流程的局限性而诞生的。瀑布模型追求可预测性并试图避免变更，而敏捷则拥抱现实——世界、市场和用户充满不确定性与不可预测性。例如，客户最初可能要求功能A，但当最终产品交付时，他们才意识到自己真正需要的是功能B。敏捷旨在通过更快速地获取客户反馈来解决这个问题，以确保团队构建的正是客户真正想要的。

秉持敏捷思维的一部分，是持续寻求更高效的工作方式。我们通过在不降低产品质量或价值的前提下，找到简化流程的方法来实现这一点。简化的关键在于减少浪费。例如，不必要的文档是一种浪费；另一种浪费是花费数周或数月开发一个功能，最终却发现客户（也可能是用户或利益相关者）根本不喜欢这个功能。

你可以通过加强团队与利益相关者之间的协作，来减少或消除这两种形式的浪费。更多的协作意味着更少的文档，以及对产品的更早期反馈。

让我们进一步探讨瀑布与敏捷之间的更多差异。项目的三个重要方面是：**需求**、**文档**和**可交付成果**。

**需求**是指为确保项目成功完成而必须满足的条件或必须完成的任务。可以将其视为项目范围内的标准集合，或必须满足的规格清单。

以下是两种模型处理需求的方式对比：

*   **瀑布模型**：在瀑布项目中，你可能需要一份详细的产品需求文档，其中列出了项目的范围和要求。你需要多个正式批准的项目计划，甚至可能有一个专门负责编写和批准这些计划的团队。你还可能设立变更控制委员会，这是一个管理需求变更的正式且严格的过程。所有这些设计都是为了保护团队，避免构建出客户或利益相关者不想要的东西，并旨在最小化可能导致范围蔓延的任何变更。当期望的最终产品已知且明确时，正式批准的项目计划效果很好，例如，领导一个基于强制性法规、具有明确要求和目标的项目。
*   **敏捷模型**：在敏捷中，需求被视为更具动态性，并预期会随着团队收到反馈和新信息而改变。项目启动时通常有一套初始需求或功能构想。但这份需求和功能清单在整个项目过程中会持续增长和变化。团队与利益相关者合作，对需求进行优先级排序，始终将最紧急或最有价值的项目移到列表顶部。然后，团队以迭代的方式处理列表中的需求。通过这种方式，团队能够在每次迭代结束时快速、频繁地获得工作反馈，并可以在继续之前对需求进行必要的调整。

接下来，我们看看**文档**。所有项目都需要文档：项目计划、利益相关者地图、时间表、章程、合同等等。

以下是两种模型处理文档的方式对比：

*   **瀑布模型**：瀑布项目使用大量文档，因为在项目阶段之间以及项目内不同团队之间存在许多交接。此外，由于工作是以较大块完成的，你需要在项目的每个阶段留下更多的文档。
*   **敏捷模型**：敏捷强调实时的、人与人之间的对话。这并不意味着完全没有文档，只是形式不同。它不再是大而正式、具有严格变更管理和审批流程的文档，而是更简短、仅包含足够细节以实现其目的的文档。这些文档更侧重于读者需要知道什么来完成工作，并且只在需要时才编写。

最后，我们来探讨**可交付成果**。可交付成果是项目产生的有形产出。

以下是两种模型处理可交付成果的方式对比：

*   **瀑布模型**：在瀑布项目中，你通常直到最后才发布可交付成果。最终的产品发布感觉像是一个大事件，伴随着重大公告和大量宣传，通常非常有趣和令人兴奋。
*   **敏捷模型**：敏捷同样令人兴奋，但它有更小、更频繁的发布。因此，每次发布的庆祝活动不那么正式，但累积起来同样令人兴奋。当项目中存在很多不确定性时（例如在新兴行业或市场中），稳定地发布项目可交付成果，使敏捷团队能够边走边获取反馈和学习。没有定期的反馈，团队最终可能会交付客户不想要的东西。

现在，你对区分敏捷与瀑布模型的一些关键要素有了更清晰的认识。这两种项目管理方法的三个主要区别在于它们各自处理**需求**、**文档**和**可交付成果**的方式。

![](img/9c4b787b2d995fdd5f625522ce6b2907_4.png)

在本节课中，我们一起学习了敏捷与瀑布模型在核心工作流程上的根本区别。我们了解到，敏捷通过拥抱变化、加强协作、减少浪费以及进行小步快跑的迭代交付，来应对项目中的不确定性。下一节视频，我们将近距离、深入地了解《敏捷宣言》及其价值观与原则。
