# 015：类模板参数推导 (CTAD) 完全指南

![](img/6d1aa3d21df92408da4341db841f07ad_0.png)

在本节课中，我们将要学习 C++17 引入的一个重要特性：类模板参数推导。我们将了解它是什么、它如何工作、它的优势以及在使用时需要注意的事项。

## 概述：什么是类模板参数推导 (CTAD)？

在 C++17 之前，创建类模板的实例时必须显式指定所有模板参数。例如，要创建一个 `std::vector<int>`，你必须写成 `std::vector<int> v`。类模板参数推导允许编译器根据你提供的初始化器，自动推导出模板参数，从而简化代码。

## 从函数模板参数推导到类模板参数推导

上一节我们介绍了 CTAD 的基本概念，本节中我们来看看它的起源。

函数模板参数推导自 C++ 诞生之初就已存在。当你调用一个函数模板时，编译器可以根据函数参数的类型推导出模板参数。

```cpp
template<typename T>
void my_func(T param) {}

my_func(1); // 编译器推导出 T 为 int
```

然而，对于类模板，情况则不同。在 C++17 之前，你必须显式指定模板参数。

```cpp
template<typename T>
struct MyStruct {};

MyStruct<int> s; // C++17 之前必须这样写
```

为了绕过这个限制，库作者们通常会编写“工厂函数”（make functions），利用函数模板的参数推导来间接实现类的推导。

```cpp
template<typename T>
MyStruct<T> make_my_struct(T t) { return MyStruct<T>(t); }

auto s = make_my_struct(1); // 推导出 MyStruct<int>
```

这种方法带来了大量的样板代码。因此，C++17 引入了类模板参数推导，允许编译器直接根据构造函数参数推导类模板的模板参数。

```cpp
MyStruct s(1); // C++17 起，编译器推导出 s 的类型是 MyStruct<int>
```

## CTAD 的优势与应用场景

了解了 CTAD 的由来后，本节我们来看看它如何让代码变得更简洁、更易读。

以下是 CTAD 带来便利的几个例子：

*   **简化标准库容器的声明**：`std::vector v{1, 2, 3};` 推导为 `std::vector<int>`。
*   **推导非类型模板参数**：`std::array arr{1, 2, 3};` 推导为 `std::array<int, 3>`。
*   **处理复杂类型**：对于 `std::pair<std::map<std::string, std::vector<int>>::iterator, bool>` 这样的长类型，使用 CTAD 可以简化为 `std::pair(it, true)`。
*   **结合新特性**：在 C++20 的 Ranges 库中，`std::ranges::join_view` 等视图类型可以无缝使用 CTAD，降低了学习新库的入门门槛。

## CTAD 的工作原理：隐式推导指南

上一节我们看到了 CTAD 的便利性，本节中我们来深入其工作原理。

CTAD 的核心机制是“推导指南”。当你写下 `MyClass obj(arg);` 时，编译器会为该类生成或查找一组“推导指南”，并尝试使用函数模板参数推导的规则来匹配这些指南，从而确定 `MyClass` 的模板参数。

对于大多数类，编译器会根据其**主模板的构造函数**自动生成“隐式推导指南”。

考虑以下类模板：

```cpp
template<typename T>
struct S {
    S(const T&);
    S(std::optional<T>);
};
```

当你写下 `S x(1);` 时，编译器会为 `S` 生成两个隐式推导指南，它们类似于以下虚构的函数模板：

```cpp
// 基于 S(const T&) 生成的指南
template<typename T> S<T> Deduce_S(const T&);
// 基于 S(std::optional<T>) 生成的指南
template<typename T> S<T> Deduce_S(std::optional<T>);
```

然后，编译器尝试用 `1` 去调用这些“函数”。第一个指南匹配成功，推导出 `T` 为 `int`。最终，`x` 被初始化为 `S<int>`。

除了基于构造函数的指南，编译器还会自动生成两个特殊的隐式推导指南：

1.  **复制推导指南**：模仿拷贝构造函数，形如 `template<typename T> S<T> Deduce_S(S<T>);`。它使得 `S y = x;` 中的 `y` 与 `x` 类型相同。
2.  **条件性推导指南**：如果类没有声明任何构造函数，则会生成一个模仿默认构造函数的指南。

## 聚合类与 CTAD

上一节我们介绍了基于构造函数的推导，本节中我们来看看没有构造函数的聚合类如何处理。

聚合类（例如简单的 `struct`）使用聚合初始化，而不是构造函数。从 C++20 开始，CTAD 也支持聚合类。

对于聚合类，编译器会生成复制推导指南和条件性推导指南。此外，当你进行聚合初始化时，编译器会**根据你提供的初始化列表**生成一个特殊的隐式推导指南，该指南尝试将初始化器的每个元素与聚合体的每个成员进行匹配和推导。

```cpp
template<typename T, typename U>
struct Aggregate {
    T t;
    U u;
    int i;
};

Aggregate a{1, ‘c‘, 3}; // C++20 起，推导出 Aggregate<int, char>
```

## 用户定义的推导指南

有时，隐式生成的推导指南无法满足需求，或者会产生我们不希望的推导。这时，我们可以编写“用户定义的推导指南”。

以下是需要用户定义推导指南的常见场景：

*   **构造函数参数无法推导模板参数**：例如 `std::vector` 的迭代器范围构造函数。迭代器的类型 `InputIt` 可以推导，但我们需要的是 `iterator_traits<InputIt>::value_type`。
*   **为特定特化提供推导**：当主模板未定义，只有特化模板有构造函数时，隐式指南无法工作，需要用户为特化提供指南。
*   **覆盖或细化推导行为**：例如，希望 `S s1(s2)` 进行转换而非复制时，可以定义一个与复制推导指南签名相同但推导结果不同的用户指南，用户指南优先级更高。

用户定义推导指南的语法如下：

```cpp
template<typename InputIt>
vector(InputIt, InputIt) -> vector<typename iterator_traits<InputIt>::value_type>;
```

**重要注意事项**：
1.  不要为标准库类型或其他不属于你的类编写推导指南，这会导致未定义行为。
2.  对于聚合类，一旦提供了用户定义的推导指南，编译器将不再为该类生成基于聚合初始化的特殊隐式指南。

## CTAD 的细节与注意事项

掌握了 CTAD 的基本用法后，本节我们探讨一些关键细节和潜在陷阱。

**CTAD 与转发引用**
在函数模板中，`T&&` 在模板参数推导时可能是转发引用。但在类模板的隐式推导指南中，如果 `T` 是类模板参数，`T&&` 就是普通的右值引用，而不是转发引用。转发引用行为仅出现在模板构造函数的推导中（此时 `T` 是构造函数自己的模板参数）。

**“全部或无”的推导**
与函数模板可以部分指定参数（如 `func<int>(3.14)`）不同，类模板参数推导是“全部或无”的。一旦你显式指定了任何一个模板参数，CTAD 就会被完全禁用。这是因为部分推导的语义（尤其是涉及默认模板参数和可变参数模板时）非常复杂且存在歧义，标准委员会尚未找到完美的解决方案。

**复制优于转换**
这是 CTAD 的一个重要设计选择。考虑以下代码：
```cpp
std::vector<int> a{1,2,3};
std::vector b(a); // b 的类型是？
```
直觉上，大多数人认为 `b` 应该和 `a` 类型相同，即 `std::vector<int>`。CTAD 通过**复制推导指南**（按值接受同类对象）优先于其他需要转换的指南，确保了这一点。如果你在泛型代码中需要“容器化”行为（即 `make_vector(a)` 得到 `vector<vector<int>>`），则不应依赖 CTAD，而应显式编写代码或使用工厂函数。

**潜在的意外推导**
如果你的库是在 C++17 之前编写的，并且用户在新编译器下使用了 CTAD，他们可能会“意外地”获得推导。如果你后续修改了库，意外地改变了推导行为，可能会导致用户代码编译失败。因此，库作者应当测试并考虑 CTAD 对用户代码的影响，必要时使用“非推导语境”来禁用特定构造函数的 CTAD，或者提供明确的用户定义指南并加以文档说明。

## 总结与最佳实践

本节课中我们一起学习了 C++17 的类模板参数推导特性。

**总结如下**：
*   **是什么**：CTAD 允许编译器根据初始化器自动推导类模板的模板参数。
*   **如何工作**：基于隐式生成或用户定义的“推导指南”，利用函数模板参数推导的变体规则。
*   **优势**：极大简化了代码，特别是使用标准库容器和新特性（如 Ranges）时。
*   **注意事项**：理解其“复制优于转换”的原则；注意在泛型代码或容器转换场景中谨慎使用；知道它无法进行部分推导。

**给开发者的建议**：
*   **对于使用者**：CTAD 是一个有用的便利特性，可以积极使用以简化代码。但在涉及容器转换或需要精确控制类型的泛型代码中，最好显式指定类型。
*   **对于库作者**：你无法阻止用户使用 CTAD。应当主动测试你的类在 CTAD 下的行为，确保推导结果是合理且符合预期的。如有必要，通过用户定义推导指南来引导正确的推导，或使用非推导语境来限制不希望的推导，并将这些设计决策清晰地记录在文档中。
